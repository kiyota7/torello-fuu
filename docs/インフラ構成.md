# AWSインフラ構成

タスクボードアプリをAWS上で動かすためのインフラ構成の概要。具体的なリソース定義や
設定値は`terraform/`ディレクトリのコードを正とするため、ここでは全体像とディレクトリ構成のみを示す。

## 全体構成図

```
[利用者のブラウザ]
        │ HTTP(80番)
        ▼
   ┌─────────────────────────────┐
   │        EC2インスタンス        │
   │  ┌───────────┐ ┌───────────┐ │
   │  │ frontend  │ │ backend   │ │
   │  │ (Nginx)   │→│(SpringBoot)│ │
   │  └───────────┘ └─────┬─────┘ │
   └───────────────────────┼──────┘
                            │ 5432番
                            │ (EC2のセキュリティグループからのみ許可)
                            ▼
                    ┌───────────────┐
                    │  RDS(PostgreSQL) │
                    │ 外部からは接続不可 │
                    └───────────────┘
```

- frontend・backendはEC2上でDocker Composeによりコンテナとして稼働する
- データベースはRDS(マネージドPostgreSQL)を利用し、EC2のセキュリティグループ経由の通信のみを許可する
  (インターネットや他のリソースからは接続できない)
- EC2にはElastic IPを割り当て、アクセス先のIPが変わらないようにしている
- EC2の操作はSSH鍵を使わず、IAMロール経由のAWS Systems Manager(SSM) Session Managerで行う
  (SSH用ポートは開放していない)
- ネットワークはAWSアカウントに最初から用意されている「デフォルトVPC」を利用し、追加のVPC構築は行っていない

## IaC(Infrastructure as Code)によるコード管理

上記のインフラはすべて**Terraform**でコード化して管理しており、マネジメントコンソールの
手動操作は行わない。構築・変更は`terraform apply`、EC2内の操作は`aws ssm send-command`など、
AWS CLIベースの操作のみで完結する。

具体的なリソース定義(インスタンスサイズやストレージ量などの設定値を含む)は、
今後変更される可能性があるため本ドキュメントには記載せず、`terraform/`配下の各`.tf`ファイルを
直接参照すること。

## ディレクトリ構成

```
.
├── terraform/               # インフラのコード化(EC2・RDS・ネットワーク・IAM等)
├── backend/Dockerfile       # backendコンテナのビルド定義
├── frontend/Dockerfile      # frontendコンテナのビルド定義
├── frontend/nginx.conf      # frontend内Nginxの配信設定・/apiのリバースプロキシ設定
├── docker-compose.prod.yml  # 本番環境でfrontend・backendコンテナをまとめて起動する定義
├── .env.example              # 本番用環境変数のサンプル(実際の値は.gitignore対象)
└── scripts/deploy.sh         # コード変更時に、AWS CLI経由でEC2上のコンテナを再デプロイするスクリプト
```

## 運用の基本方針

- インフラそのものの新規構築・変更: `terraform apply`
- アプリのコード変更のみを反映する再デプロイ: `scripts/deploy.sh`
- 環境の削除(コスト停止): `terraform destroy`

いずれもAWSマネジメントコンソールやSSHを使わず、AWS CLIベースの操作で行う方針としている。
